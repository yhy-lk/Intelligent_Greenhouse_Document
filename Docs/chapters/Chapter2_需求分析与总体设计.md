# 第2章 需求分析与总体设计

## 2.1 系统需求分析

### 2.1.1 功能需求

根据温室环境管理的实际作业场景，本系统的功能需求归纳为以下五个方面：

（1）**环境数据采集**。系统需实时采集温室内的空气温度、空气湿度、土壤湿度和光照强度。温度与湿度通过 SHT30 数字温湿度传感器获取（I2C 接口，温度精度 ±0.3°C，湿度精度 ±2%RH）[@sht30datasheet]；光照通过 BH1750 数字光照传感器获取（量程 1～65535 Lux）[@bh1750datasheet]；土壤湿度通过模拟量传感器经 ADC 采样获取。

（2）**执行器控制**。系统需对通风风扇（PWM 调速，支持 PID 闭环转速控制）、水泵（开关控制）、加湿器（开关控制）、补光灯（WS2812B RGB 灯带）和遮阳舵机（MG995）五种执行器进行精确控制，各执行器均需支持手动操控和自动调度两种工作方式。

（3）**三种控制模式**。系统需支持手动控制、自动控制和 AI 领航三种控制模式。手动模式由用户通过主节点触摸屏直接操控从节点执行器；自动模式由各从节点独立运行本地迟滞调度器，根据预设阈值自动调节执行器；AI 领航模式通过集成 DeepSeek 大语言模型，支持语音交互驱动的智能环境分析与决策建议。

（4）**用户交互**。系统需提供图形界面与语音交互两种人机交互方式。图形界面基于 LVGL v9.3.0 框架[@lvgl2024]，运行于 240×320 像素的 TFT 彩色触摸屏上，包含主页、总览页、控制页等 8 个页面。语音交互通过 INMP441 麦克风采集语音，经百度 STT 转文本后送入 DeepSeek 推理，推理结果经百度 TTS 转语音输出，形成完整的交互链路。

（5）**分布式数据通信**。系统需在主节点与多个从节点之间建立可靠的通信通道，支持从节点周期性上报传感器数据、主节点下发控制命令与参数配置、异常告警及时间同步等功能。

### 2.1.2 性能需求

（1）**传感器采样周期**。SHT30、BH1750 和土壤湿度 ADC 的采样周期均为 1 秒（1 Hz），为自动调度器提供及时的输入数据。

（2）**控制响应时间**。本地自动调度器的调度周期为 500 毫秒（2 Hz），通风风扇的 PID 转速闭环控制周期为 100 毫秒（10 Hz）。CAN 总线在 1 Mbps 波特率下通信延迟小于 1 毫秒，不构成系统响应瓶颈。

（3）**CAN 总线通信速率**。系统采用 1 Mbps 的 CAN 总线通信速率，CAN 2.0A 标准帧的 11-bit 标识符提供 4-bit 功能码与 7-bit 节点 ID 的组合寻址能力，理论上支持最多 127 个从节点。

（4）**系统稳定性**。系统需支持 7×24 小时连续运行。ESP32 主节点通过 FreeRTOS 任务调度实现多任务并发，STM32 从节点通过 Embassy 异步运行时实现非阻塞调度，主节点每 60 秒监控堆内存使用情况，从节点通过硬件看门狗确保异常时自动复位。

### 2.1.3 非功能需求

（1）**模块化与可扩展性**。硬件层面通过 CAN 总线实现节点即插即用扩展，软件层面通过分层架构实现功能模块的独立开发与测试。新增传感器或执行器只需在对应节点的驱动层添加模块，无需修改系统架构。

（2）**低功耗设计**。传感器与执行器均选用低功耗器件，CAN 总线采用事件驱动的通信方式，从节点仅在数据变化或定时上报时发送帧，减少不必要的总线活动。

（3）**故障自检与容错**。从节点通过心跳机制（参数索引 0x01）向主节点报告存活状态，主节点通过超时检测（默认 5 分钟无报文判定节点离线）实现故障自动发现。STM32 侧硬件过滤器仅接收广播帧和本节点地址帧，降低无关帧处理开销。

## 2.2 系统总体架构设计

### 2.2.1 分布式多节点架构设计

为解决传统单 MCU 架构在扩展性与实时性方面的不足，本系统采用"ESP32-S3 主节点 + 多个 STM32F407 从节点"的分布式多节点架构。该架构将系统功能划分为交互层与控制层两个层次，分别由不同类型的微控制器承担，通过 CAN 总线实现跨节点的数据交换与协调控制。系统总体架构如图 2-1 所示。

**图 2-1 分布式多节点系统总体架构**

![图 2-1 分布式多节点系统总体架构](../images/system_architecture.png)

主节点（ESP32-S3，主频 240 MHz，双核 Xtensa LX7）承担交互层功能：LVGL 图形界面渲染与事件处理、语音助手录音与播放、DeepSeek 大语言模型 API 调用与推理、WiFi 网络连接管理，以及多节点状态聚合管理[@esp32techref]。从节点（STM32F407VET6，主频 168 MHz，ARM Cortex-M4F，内置 FPU）承担控制层功能：传感器数据采集、本地自动调度（迟滞控制）、PID 闭环控制（通风风扇转速）以及执行器驱动。

该分布式架构具有以下优势：任务解耦——ESP32 专注于网络通信与 AI 推理等计算密集型任务，STM32 专注于传感器采集与实时控制等时间敏感型任务；可扩展性——CAN 总线原生支持多主多从拓扑，新增温室区域只需部署新的 STM32 从节点接入总线；可靠性——单个从节点故障不影响其他节点运行，各从节点独立维护本地控制逻辑；实时性——STM32 本地闭环控制延迟小于 10 毫秒，无需等待主节点指令即可完成执行器调节[@shamshiri2018advances]。

### 2.2.2 CAN 总线多节点通信架构

本系统的主从节点间通信基于 CAN 2.0A 标准帧实现。CAN 总线采用总线型拓扑，主节点与所有从节点共享 CAN_H/CAN_L 差分信号线，总线两端各连接 120Ω 终端电阻，通信波特率为 1 Mbps。CAN 帧格式设计如表 2-1 所示。11-bit 标准标识符划分为 4-bit 功能码（Bit 10..7）与 7-bit 节点 ID（Bit 6..0），构建公式为 $\text{CAN\_ID} = (\text{func\_code} \ll 7) \, | \, \text{node\_id}$。8-byte 数据域采用"参数索引 + 保留 + 参数值"的统一格式。

**表 2-1 CAN 帧格式设计**

| 字段 | 位置 | 长度 | 说明 |
|:---:|:---:|:---:|:---|
| 功能码 | Bit 10..7 | 4-bit | 帧的业务类型（Alert/TimeSync/WriteSet/Report） |
| 节点 ID | Bit 6..0 | 7-bit | 目标或来源节点地址（0 为广播，1-127 为从节点） |
| 参数索引 | Byte 0 | 1-byte | 参数字典中的索引编号 |
| 保留字段 | Byte 1-3 | 3-byte | 预留未来扩展 |
| 参数值 | Byte 4-7 | 4-byte | 32-bit 数据值，小端序存储 |

功能码按优先级从高到低分为四类：Alert（0x0，从→主，异常告警，最高优先级）、TimeSync（0x1，主→从，时间同步广播）、WriteSet（0x2，主→从，控制命令与参数配置）、Report（0x3，从→主，传感器数据上报，最低优先级）。参数字典覆盖系统控制所需的全部参数，温度与湿度参数采用 ×100 缩放因子（如 25.5°C 传输为 2550），PID 参数采用 ×1000 缩放因子。STM32 从节点通过硬件过滤器配置两组 Mask32 过滤器，分别匹配广播地址和本节点地址，从硬件层面过滤无关帧。

### 2.2.3 分层软件架构

为实现模块化设计与关注点分离，主节点与从节点分别采用不同的分层软件架构。

**ESP32 主节点分层架构**如图 2-2 所示，采用四层嵌入式软件架构：App 应用层负责任务编排；Service 服务层承载核心业务逻辑，包含 SensorState（多节点状态管理）、CanService（CAN 协议编解码）、GUI Manager（LVGL 界面管理）、DeepSeekAPI（大语言模型集成）、VoiceAssistant（语音助手状态机）、WifiService（WiFi 连接管理）等模块；ECUAL 层提供 ECU 级硬件抽象；MCAL 层提供控制器级外设抽象[@zhu2020embedded]。

**图 2-2 ESP32 主节点分层软件架构**

![图 2-2 ESP32 主节点分层软件架构](../images/esp32_layered_architecture.png)

**STM32 从节点分层架构**如图 2-3 所示，采用 Rust 语言与 Embassy 异步运行时框架[@embassy2024]，按 Cargo crate 组织为四层结构：firmware 固件层包含 main.rs 与 shared.rs（全局状态单例 `GLOBAL_STATE`，由 `CriticalSectionRawMutex` 保护）以及按功能域组织的异步任务；crates/app 应用层封装传感器数据读取与执行器高级控制逻辑；crates/bsw 基础软件层提供 can_proto（CAN 协议编解码）、pid（离散位置式 PID 控制器）、motor_ctrl（风扇电机闭环控制）、sht30 和 bh1750（I2C 传感器驱动）等模块；crates/service 服务层提供颜色转换等公共服务。

**图 2-3 STM32 从节点分层软件架构**

两个节点的软件架构虽然在语言（C/C++ vs Rust）和运行时（FreeRTOS vs Embassy）上不同，但均遵循"分层解耦、逐层调用"的设计原则，CAN 总线协议作为主从节点间的唯一通信接口，实现跨节点的松耦合集成。

## 2.3 控制策略设计

本系统设计了手动控制、自动控制和 AI 领航三种控制模式，逻辑关系如图 2-4 所示。

![图 2-4 三种控制模式逻辑关系](../images/control_strategy.png)

### 2.3.1 手动控制模式

手动控制模式下，用户通过主节点触摸屏 GUI 直接操控指定从节点的执行器。操作流程为：用户在控制页选择目标从节点，进入手动模式页后，通过开关控件控制遮阳舵机、水泵、加湿器等开关型执行器的启停，通过滑动条调节通风风扇转速和补光灯亮度与颜色。操作事件由 LVGL 事件回调捕获后，通过 CAN 总线以 WriteSet 功能码帧发送至目标从节点，从节点解析帧后直接更新执行器状态。该模式适用于设备调试、紧急停机等需要人工干预的场景。

### 2.3.2 自动控制模式

自动控制模式下，各 STM32 从节点独立运行本地迟滞调度器，根据传感器数据与预设阈值自动控制执行器状态。迟滞控制（Hysteresis Control）是一种带死区的开关控制策略，通过设置开启阈值偏移与关闭阈值偏移形成迟滞带，避免被控量在阈值附近频繁波动导致执行器反复启停[@hu2014automatic]。

调度器的调度周期为 500 毫秒，每个周期从全局状态 `GLOBAL_STATE` 中读取最新传感器数据与目标阈值。以温度控制为例，当温度超过目标值加上开启偏移量（`TEMP_FAN_ON_MARGIN_C = 1.0°C`）时启动风扇，当温度降至目标值加上关闭偏移量（`TEMP_FAN_OFF_MARGIN_C = 0.3°C`）以下时停止风扇，形成 0.7°C 的迟滞带。风扇启动后，转速根据温度偏差和湿度偏差的比例关系动态计算，映射区间为 1200 RPM 至 4500 RPM。加湿器和水泵分别采用类似的低侧迟滞策略，补光灯则采用比例控制方式，每次调度周期递增或递减 1% 实现平滑亮度过渡。

用户通过主节点自动模式页配置各参数的目标阈值，阈值通过 CAN 总线以 WriteSet 功能码下发至从节点。该设计使得各从节点在与主节点通信中断时，仍可基于本地存储的阈值继续执行自动控制，保障了系统的鲁棒性。

### 2.3.3 AI 领航控制模式

AI 领航模式通过集成 DeepSeek 大语言模型实现语音交互驱动的智能环境分析。交互流程为：用户发出语音指令后，系统依次经过语音录音、语音识别（百度 STT）、大模型推理（DeepSeek Chat API）、语音合成（百度 TTS）和语音播放五个阶段。语音助手采用六状态有限状态机设计，包括 WAITING、RECORDING、PROCESSING_STT、PROCESSING_LLM、PROCESSING_TTS 和 PLAYBACK 六个状态。

DeepSeek 大模型的集成采用上下文感知的提示词构建策略。每次推理前，系统通过 SensorState 模块获取目标从节点的实时传感器数据快照，结构化注入 LLM 上下文中，使模型能够基于当前环境状态进行分析。模型配置为 `deepseek-chat`，温度参数 0.7，最大输出 180 token，对话历史窗口 6 条消息。

需要说明的是，当前版本的 AI 领航模式仅支持信息查询与建议生成，不具备直接控制执行器的能力。LLM 推理结果以文本形式展示在 AI 领航页面上，同时通过语音合成播放给用户。用户可根据 LLM 建议手动切换至其他模式进行环境调控。该模式将大语言模型的自然语言理解能力与嵌入式传感器数据相结合，为温室管理提供了智能辅助决策范式[@tzachor2023llm; @shaikh2025llm]。
