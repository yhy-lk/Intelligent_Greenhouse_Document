# 第2章 需求分析与总体设计

## 2.1 系统需求分析

在系统设计之初，需要对智能温室控制系统的功能需求、性能需求与非功能需求进行全面分析，以明确系统边界、指导后续的硬件选型与软件架构设计。本节从温室环境调控的实际应用场景出发，结合分布式嵌入式系统的设计约束，对系统需求进行逐层剖析。

### 2.1.1 功能需求

根据温室环境管理的实际作业场景，本系统的功能需求可归纳为以下五个方面：

（1）**环境数据采集**。系统需实时采集温室内的多维度环境参数，包括空气温度、空气湿度、土壤湿度和光照强度。温度与湿度数据通过 SHT30 数字温湿度传感器获取，该传感器采用 I2C 总线接口，温度测量精度为 ±0.3°C，湿度测量精度为 ±2%RH[@sht30datasheet]；光照强度通过 BH1750 数字光照传感器获取，量程为 1～65535 Lux[@bh1750datasheet]；土壤湿度通过模拟量传感器经 ADC 采样获取。上述传感器数据是自动控制与智能决策的基础数据源。

（2）**执行器控制**。系统需对温室内的多种执行器进行精确控制，包括：通风风扇（PWM 调速，支持 PID 闭环转速控制）、水泵（开关控制，用于灌溉）、加湿器（开关控制，用于调节空气湿度）、补光灯（WS2812B RGB 灯带，支持亮度与颜色调节）以及遮阳舵机（MG995，开关式遮阳控制）。各执行器均需支持手动操控和自动调度两种工作方式。

（3）**三种控制模式**。系统需支持手动控制、自动控制和 AI 领航三种控制模式，以满足不同场景下的管理需求。手动模式由用户通过主节点触摸屏直接操控指定从节点的执行器；自动模式由各从节点独立运行本地迟滞调度器，根据预设阈值自动调节执行器状态；AI 领航模式通过集成 DeepSeek 大语言模型，支持语音交互驱动的智能环境调控，能够根据多节点传感器数据进行综合分析并给出决策建议。

（4）**用户交互**。系统需提供图形界面显示与语音交互两种人机交互方式。图形界面基于 LVGL v9.3.0 框架开发[@lvgl2024]，运行于 240×320 像素的 TFT 彩色触摸屏上，包含主页、总览页、控制页、手动模式页、自动模式页和 AI 领航页等 8 个页面，支持实时数据显示、执行器操控与阈值配置。语音交互通过 INMP441 麦克风采集语音，经百度语音识别（STT）转换为文本后送入 DeepSeek 大语言模型进行推理，推理结果再经百度语音合成（TTS）转换为语音输出，形成完整的"语音输入—大模型推理—语音输出"交互链路。

（5）**分布式数据通信**。系统需在主节点与多个从节点之间建立可靠的数据通信通道。通信内容包括从节点周期性上报传感器数据、主节点下发控制命令与参数配置、异常告警以及时间同步等。通信协议需支持多节点寻址，具备一定的容错能力。

### 2.1.2 性能需求

为确保系统在实际温室环境中的可靠运行，本系统在实时性、通信速率和稳定性等方面需满足以下性能指标：

（1）**传感器采样周期**。温湿度传感器（SHT30）的采样周期为 1 秒（1 Hz），光照传感器（BH1750）的采样周期为 1 秒，土壤湿度传感器的 ADC 采样周期同样为 1 秒。该采样频率能够满足温室环境缓慢变化的特征，同时为自动调度器提供及时的输入数据。

（2）**控制响应时间**。本地自动调度器的调度周期为 500 毫秒（2 Hz），即从传感器数据更新到执行器状态切换的最大延迟不超过 500 毫秒。通风风扇的 PID 转速闭环控制周期为 100 毫秒（10 Hz），能够实现快速的转速调节。CAN 总线通信延迟在 1 Mbps 波特率下通常小于 1 毫秒，远低于控制周期，不会成为系统响应的瓶颈。

（3）**CAN 总线通信速率**。系统采用 1 Mbps 的 CAN 总线通信速率，该速率在短距离总线（< 40 米）条件下能够保证可靠的多节点通信。CAN 2.0A 标准帧的 11-bit 标识符提供了 4-bit 功能码与 7-bit 节点 ID 的组合寻址能力，理论上支持最多 127 个从节点。

（4）**系统稳定性**。系统需支持 7×24 小时连续运行。ESP32 主节点通过 FreeRTOS 任务调度实现多任务并发执行，STM32 从节点通过 Embassy 异步运行时实现非阻塞的任务调度，两者均需具备内存泄漏检测与异常恢复机制。主节点每 60 秒监控一次堆内存使用情况，从节点通过硬件看门狗确保系统异常时能够自动复位。

### 2.1.3 非功能需求

除功能与性能需求外，系统还需满足以下非功能需求：

（1）**模块化与可扩展性**。系统采用模块化设计，硬件层面通过 CAN 总线实现节点的即插即用扩展，软件层面通过分层架构实现功能模块的独立开发与测试。新增传感器类型或执行器只需在对应节点的驱动层添加模块，无需修改系统架构。

（2）**低功耗设计**。传感器与执行器均选用低功耗器件，CAN 总线采用事件驱动的通信方式（从节点仅在数据变化或定时上报时发送帧），减少不必要的总线活动。

（3）**故障自检与容错机制**。从节点通过心跳机制（Heartbeat，参数索引 0x01）向主节点报告存活状态，主节点通过超时检测（默认 5 分钟无报文判定节点离线）实现节点故障的自动发现。CAN 总线的 CRC 校验机制保证数据传输的完整性，STM32 侧的硬件过滤器仅接收广播帧和本节点地址帧，减少无关帧的处理开销。

## 2.2 系统总体架构设计

### 2.2.1 分布式多节点架构设计

为解决传统单 MCU 架构在扩展性与实时性方面的不足，本系统采用"ESP32-S3 主节点 + 多个 STM32F407 从节点"的分布式多节点架构。该架构将系统功能划分为交互层与控制层两个层次，分别由不同类型的微控制器承担，通过 CAN 总线实现跨节点的数据交换与协调控制。系统总体架构如图 2-1 所示。

**图 2-1 分布式多节点系统总体架构**

![图 2-1 分布式多节点系统总体架构](../images/system_architecture.png)

主节点（ESP32-S3）承担交互层的全部功能，具体包括：LVGL 图形界面的渲染与事件处理、语音助手的录音与播放、DeepSeek 大语言模型的 API 调用与推理、WiFi 网络连接管理，以及多节点状态的聚合管理。ESP32-S3 主频为 240 MHz，采用双核 Xtensa LX7 架构，内置 WiFi 与蓝牙 5.0 模块[@esp32techref]，其丰富的外设接口（SPI、I2C、I2S、UART）和双核并行处理能力使其适合承担高并发的交互层任务。

从节点（STM32F407VET6）承担控制层的全部功能，具体包括：传感器数据采集（SHT30 温湿度、BH1750 光照、土壤湿度 ADC）、本地自动调度（迟滞控制）、PID 闭环控制（通风风扇转速）以及执行器驱动（PWM、GPIO、继电器）。STM32F407 主频为 168 MHz，采用 ARM Cortex-M4F 架构，内置浮点运算单元（FPU），支持 DSP 指令集，其丰富的定时器资源（TIM1 用于 PWM 输出、TIM3 用于正交编码器接口）使其适合承担实时控制任务。

该分布式架构具有以下优势：

（1）**任务解耦**。交互层与控制层分离，降低了单节点的软件复杂度。ESP32 专注于网络通信、图形渲染与 AI 推理等计算密集型任务，STM32 专注于传感器采集与实时控制等时间敏感型任务，两者各司其职，互不干扰。

（2）**可扩展性**。CAN 总线原生支持多主多从拓扑，7-bit 节点 ID 可寻址 127 个从节点（节点 ID 0 保留为广播地址）。新增温室区域只需部署新的 STM32 从节点并接入 CAN 总线，主节点通过 SensorState 模块自动管理新节点的状态数据（最多支持 16 个节点的实时状态跟踪）。

（3）**可靠性**。单个从节点的故障不影响其他节点的正常运行。每个从节点独立维护本地控制逻辑，即使与主节点的通信中断，仍可基于本地传感器数据继续执行自动控制策略。

（4）**实时性**。STM32 从节点的本地闭环控制延迟小于 10 毫秒（PID 控制周期 100 毫秒），无需等待主节点的指令即可完成执行器的快速调节，满足温室环境控制对实时性的要求[@shamshiri2018advances]。

### 2.2.2 CAN 总线多节点通信架构

本系统的主从节点间通信基于 CAN 2.0A 标准帧实现。CAN 总线采用总线型拓扑结构，主节点与所有从节点共享 CAN_H / CAN_L 差分信号线，总线两端各连接 120Ω 终端电阻以匹配阻抗，通信波特率为 1 Mbps。CAN 总线的非破坏性仲裁机制确保高优先级帧（如异常告警）能够在总线竞争中优先传输，满足分布式控制系统对通信实时性的要求[@niveditha2018can]。

CAN 帧格式设计如表 2-2 所示。11-bit 标准标识符被划分为 4-bit 功能码（Bit 10..7）与 7-bit 节点 ID（Bit 6..0）两个字段，构建公式为 $\text{CAN\_ID} = (\text{func\_code} \ll 7) \, | \, \text{node\_id}$。功能码定义了帧的业务类型，节点 ID 标识了帧的目标（或来源）节点。8-byte 固定长度的数据域采用"参数索引 + 保留 + 参数值"的统一格式：Byte 0 为参数索引（1 字节），Byte 1-3 为保留字段（3 字节，必须为 0x00），Byte 4-7 为参数值（4 字节，小端序）。

**表 2-2 CAN 帧格式设计**

| 字段 | 位置 | 长度 | 说明 |
|:---:|:---:|:---:|:---|
| 功能码 | Bit 10..7 | 4-bit | 帧的业务类型（Alert/TimeSync/WriteSet/Report） |
| 节点 ID | Bit 6..0 | 7-bit | 目标或来源节点地址（0 为广播，1-127 为从节点） |
| 参数索引 | Byte 0 | 1-byte | 参数字典中的索引编号 |
| 保留字段 | Byte 1-3 | 3-byte | 必须为 0x00，预留未来扩展 |
| 参数值 | Byte 4-7 | 4-byte | 32-bit 数据值，小端序存储 |

功能码按优先级从高到低分为四类，如表 2-1 所示。

**表 2-1 CAN 功能码分类**

| 功能码 | 编码 | 方向 | 说明 | 优先级 |
|:---:|:---:|:---:|:---|:---:|
| Alert | 0x0 | 从→主 | 异常告警 | 最高 |
| TimeSync | 0x1 | 主→从 | 时间同步广播 | 高 |
| WriteSet | 0x2 | 主→从 | 控制命令与参数配置 | 中 |
| Report | 0x3 | 从→主 | 传感器数据上报 | 低 |

参数字典覆盖了系统控制所需的全部参数，按功能分为四组：系统参数（0x00-0x0F，含时间戳、心跳、错误码、控制模式）、开关型执行器（0x10-0x2F，含补光灯、水泵、加湿器、风扇、遮阳舵机等，值为 0 或 1）、模拟量传感器（0x30-0x4F，含温度、空气湿度、土壤湿度、光照强度、风扇转速等，采用缩放因子将浮点物理量映射为整数传输）以及高级参数（0x50-0x5F，含 RGB 颜色、PWM 占空比、PID 参数）。温度与湿度参数采用 ×100 的缩放因子（如 25.5°C 传输为 2550），PID 参数采用 ×1000 的缩放因子，以在 32-bit 整数范围内保留足够的精度。

STM32 从节点通过硬件过滤器实现帧的地址过滤：配置两组 Mask32 过滤器，分别匹配广播地址（0x000）和本节点地址（NODE_ID = 0x01），仅接收目标节点 ID 为广播地址或本节点地址的帧，从硬件层面过滤无关帧，降低 CPU 负载。

### 2.2.3 分层软件架构

为实现模块化设计与关注点分离，系统的主节点与从节点分别采用不同的分层软件架构。

**ESP32 主节点分层架构**如图 2-3 所示，采用经典的四层嵌入式软件架构：App 应用层负责任务编排，包含 CAN App（初始化 CAN 网络服务并启动轮询任务）和 LVGL App（初始化显示驱动并启动界面事件处理与刷新任务）；Service 服务层承载核心业务逻辑，包含 SensorState（多节点数字孪生状态管理）、CanService（CAN 协议编解码与硬件抽象）、GUI Manager（LVGL 界面管理与事件分发）、DeepSeekAPI（大语言模型集成）、VoiceAssistant（语音助手状态机）、WifiService（WiFi 自动连接与多凭证管理）等 10 个服务模块；ECUAL 层提供 ECU 级硬件抽象，包括显示驱动（240×320 并行 8080 总线 + SPI 触摸）、INMP441 麦克风驱动和 MAX98357A 功放驱动；MCAL 层提供控制器级外设抽象（I2S、WiFi HAL）；BSP 层提供板级支持（FreeRTOS 任务栈与优先级配置）[@zhu2020embedded]。

## 2.3 控制策略设计

本系统设计了手动控制、自动控制和 AI 领航三种控制模式，以满足不同场景下温室环境管理的需求。三种模式的逻辑关系如图 2-5 所示。

![图 2-5 三种控制模式逻辑关系](../images/control_strategy.png)

### 2.3.1 手动控制模式

手动控制模式下，用户通过主节点的触摸屏 GUI 直接操控指定从节点的执行器。操作流程为：用户在控制页选择目标从节点，进入手动模式页后，通过开关控件控制遮阳舵机、水泵、加湿器等开关型执行器的启停，通过滑动条控件调节通风风扇的转速（目标 RPM）和补光灯的亮度（PWM 占空比）与颜色（HSV 色彩空间转换为 RGB 值）。用户的操作事件由 LVGL 事件回调函数捕获后，通过 CAN 总线以 WriteSet 功能码（0x2）帧发送至目标从节点，从节点的 CAN 接收任务解析帧后直接更新本地执行器状态。

手动模式适用于需要人工干预的特殊场景，如设备调试、紧急停机或精细化管理。在手动模式下，从节点的本地自动调度器不介入执行器控制，所有控制权交由主节点远程操控。

### 2.3.2 自动控制模式

自动控制模式下，各 STM32 从节点独立运行本地迟滞调度器，根据传感器数据与预设阈值自动控制执行器状态，无需主节点的实时干预。该模式的核心是迟滞控制（Hysteresis Control）策略，其本质是一种带死区的开关控制，通过设置"开启阈值偏移"与"关闭阈值偏移"形成迟滞带，避免被控量在阈值附近频繁波动导致执行器反复启停[@hu2014automatic]。

迟滞调度器的调度周期为 500 毫秒，每个周期从本地全局状态（`GLOBAL_STATE`）中读取最新的传感器数据与目标阈值，根据以下策略控制各执行器：

（1）**通风风扇控制**（高侧迟滞）。当温度超过目标值加上开启偏移量（1.0°C）时启动风扇，当温度降至目标值加上关闭偏移量（0.3°C）以下时停止风扇。类似地，当空气湿度超过目标值加上开启偏移量（8.0%）时也启动风扇。风扇启动后，其转速根据温度偏差和湿度偏差的比例关系动态计算，映射区间为 1200 RPM（最低）至 4500 RPM（最高）。

（2）**加湿器控制**（低侧迟滞）。当空气湿度降至目标值减去开启偏移量（5.0%）以下时启动加湿器，当湿度回升至目标值减去关闭偏移量（2.0%）以上时停止加湿器。

（3）**水泵控制**（低侧迟滞）。当土壤湿度降至目标值减去开启偏移量（5.0%）以下时启动水泵，当土壤湿度回升至目标值减去关闭偏移量（2.0%）以上时停止水泵。

（4）**补光灯控制**。当光照强度低于目标值时，补光灯以比例控制方式调节 PWM 占空比，每次调度周期递增或递减 1% 实现平滑的亮度过渡，避免光照突变对作物造成应激。

用户通过主节点自动模式页的双范围滑动条配置各参数的目标阈值范围，阈值通过 CAN 总线以 WriteSet 功能码下发至从节点，从节点的调度器读取更新后的阈值执行自动控制。该设计使得各从节点能够在与主节点通信中断的情况下，仍基于本地存储的阈值继续执行自动控制，保障了系统的鲁棒性。

### 2.3.3 AI 领航控制模式

AI 领航模式是本系统的特色功能，通过集成 DeepSeek 大语言模型实现语音交互驱动的智能环境调控。该模式的交互流程为：用户进入 AI 领航页后，语音助手自动启动，进入等待状态；用户发出语音指令后，系统依次经过语音录音、语音识别（百度 STT）、大模型推理（DeepSeek Chat API）、语音合成（百度 TTS）和语音播放五个阶段，形成完整的交互闭环。

语音助手采用六状态有限状态机设计，状态包括：WAITING（监听麦克风，检测语音活动）、RECORDING（录制语音至 PSRAM 缓冲区）、PROCESSING_STT（调用百度语音识别 API）、PROCESSING_LLM（调用 DeepSeek 大语言模型 API）、PROCESSING_TTS（调用百度语音合成 API）和 PLAYBACK（通过 MAX98357A 功放播放合成语音）。语音活动检测基于 256 采样帧的平均绝对幅度判断，阈值设为 500；静音超时设为 1500 毫秒；录制时长低于 0.8 秒的短促噪声会被自动过滤。

DeepSeek 大语言模型的集成采用了上下文感知的提示词构建策略。在每次推理前，系统通过 SensorState 模块获取目标从节点的实时传感器数据快照（包括温度、空气湿度、土壤湿度、光照强度等），将这些数据结构化注入 LLM 的上下文中，使模型能够基于当前环境状态进行分析与决策。模型配置为 `deepseek-chat`，温度参数设为 0.7，最大输出 token 数为 180，对话历史窗口为 6 条消息，以平衡推理质量与响应速度。模型的角色设定为"温室管理专家"，输出要求为适合直接语音播放的纯文本格式。

值得注意的是，当前版本的 AI 领航模式仅支持信息查询与建议生成，尚不具备直接控制执行器的能力。LLM 的推理结果以文本形式展示在 AI 领航页面上，同时通过语音合成播放给用户，为用户提供环境分析报告和管理建议。用户可根据 LLM 的建议，手动切换至手动模式或自动模式进行相应的环境调控操作。该模式将大语言模型的自然语言理解能力与嵌入式传感器数据相结合，为温室管理提供了一种全新的智能辅助决策范式[@tzachor2023llm; @shaikh2025llm]。
