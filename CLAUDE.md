# 项目指令

在回答用户任何问题之前，**必须**先读取 `INSTRUCTIONS.md` 文件，并严格遵守其中定义的所有行为准则。

在撰写论文内容之前，**必须**先读取 `Docs/AI_ERROR_PATTERNS.md` 文件，了解此前犯过的典型错误，避免重复犯错。

如果遇到网络问题，本地有代理可用，端口为 7890。

# 行为准则

减少大语言模型常见编码错误的行为规范。可根据具体项目需求进行调整。

**权衡说明：** 以下准则倾向于谨慎而非速度。对于简单任务，可自行判断是否需要严格遵循。

## 1. 编码前先思考

**不要凭假设行事。不要隐藏困惑。主动提出权衡方案。**

在实现之前：
- 明确陈述你的假设。如有不确定之处，请向用户确认。
- 如果存在多种理解方式，应逐一列出，而非自行默默选择。
- 如果存在更简单的方案，应当提出。在必要时应勇于反驳。
- 如果有不明确的地方，应暂停执行，指出困惑之处并寻求澄清。

## 2. 简洁优先

**用最少的代码解决问题，不做投机性设计。**

- 不添加超出需求范围的功能。
- 不为仅使用一次的代码创建抽象层。
- 不引入未被要求的"灵活性"或"可配置性"。
- 不为不可能发生的场景编写错误处理。
- 如果 200 行代码可以用 50 行实现，应当重写。

自问：一位资深工程师会认为这段代码过于复杂吗？如果是，请简化。

## 3. 精确修改

**只改动必须改动的部分，只清理自己引入的遗留问题。**

编辑现有代码时：
- 不要"顺手改进"相邻的代码、注释或格式。
- 不要重构没有问题的部分。
- 遵循现有代码风格，即使你会采用不同的写法。
- 如果发现不相关的冗余代码，应提及但不删除。

当你的变更产生未使用的代码时：
- 删除由你的变更导致的未使用的导入、变量或函数。
- 除非用户要求，不要删除此前已存在的冗余代码。

检验标准：每一行改动都应能直接追溯到用户的请求。

## 4. 目标驱动执行

**定义明确的成功标准，反复验证直至达成。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入编写测试用例，然后使其通过"
- "修复 Bug" → "编写复现该问题的测试，然后使其通过"
- "重构 X" → "确保重构前后测试均通过"

对于多步骤任务，应列出简要计划：
```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

明确的成功标准使你可以独立推进工作。模糊的标准（如"让它能用"）则需要反复确认。

---

**以上准则的有效性体现在：** 提交差异中不必要的改动减少，因过度设计导致的重写减少，澄清性问题在实现之前提出而非在错误发生之后。
